# 1. System Overview

WOCS is a **multi-warehouse, multi-client operations management platform**. It replaces fragmented spreadsheets, email, and ad-hoc tools with a single centralized system covering inventory, order lifecycle, SLA tracking, billing data generation, and audit trails.

---

# 1.1 High-Level Architecture

WOCS follows a **three-tier architecture**:

1. **Frontend Layer**: Web application (React) + Mobile/Handheld PWA  
2. **Backend Layer**: RESTful API server with business logic and rule engine  
3. **Data Layer**: Relational database (PostgreSQL) + audit event store + file storage  

### Key architectural principles

1. **Configuration-over-code**  
   All per-warehouse and per-client rules are stored in the database and evaluated at runtime — no code deployments for new warehouses or clients.

2. **Immutable audit trail**  
   Every state change is appended to an audit log. Records are never deleted, only archived.

3. **Role enforcement at API layer**  
   All permissions are validated server-side. The frontend UI may hide features, but the API enforces all role-based restrictions independently.

4. **Offline resilience (bounded)**  
   Transactions buffered locally for up to 5 minutes of network loss. Full offline mode is out of scope for Phase 1.

---

# 1.2 Technology Stack

| Layer | Technology | Justification |
|---|---|---|
| Frontend (Web) | React 18 + TypeScript + Vite | Component reuse, strong typing, fast builds |
| Frontend (Mobile/Handheld) | Progressive Web App (PWA) | No app store; works on handheld barcode scanners |
| Backend API | Node.js + Express (or NestJS) | Fast JSON APIs; large ecosystem for logistics integrations |
| Database | PostgreSQL 15+ | ACID compliance, JSONB for flexible config, row-level security |
| ORM | Prisma or TypeORM | Schema migrations, type-safe queries |
| Auth | JWT + Refresh Tokens | Stateless, scalable across warehouses |
| File Storage | S3-compatible (AWS S3 or MinIO on-prem) | Bulk import files, export reports |
| Background Jobs | BullMQ (Redis-backed) | SLA monitoring cron, notification dispatch, billing calc |
| Notifications | Nodemailer (email) + Firebase (push) | Multi-channel; configurable per warehouse |
| API Gateway | Nginx reverse proxy | Rate limiting, SSL termination, path routing |
| Containerization | Docker + Docker Compose | Consistent deployments across 6–9 warehouses |
| CI/CD | GitHub Actions | Automated testing + deployment pipeline |

---

# 1.3 Deployment Topology

Single central server (cloud or on-prem data center) with:

1. One PostgreSQL instance (primary + replica for read scaling)  
2. Redis instance for job queues and session caching  
3. All warehouses connect via internet (HTTPS).  
4. Warehouse-local clients are thin web/PWA clients only.  
5. Barcode scanner devices connect via WiFi to the PWA interface.  
6. File uploads go to S3-compatible object storage.

# 2. Stakeholders & User Roles

There are **7 defined roles in WOCS**. Each role maps to a set of permissions enforced at the API level.

Role assignment is managed by **IT Administrator**.

---

# 2.1 Role Definitions

| Role | Primary Function | Key Permissions |
|---|---|---|
| Warehouse Operator | Day-to-day floor operations | Receive stock, update order states, scan barcodes |
| Warehouse Manager | Site-level oversight | Approve adjustments, manage operator accounts |
| Regional Operations Head | Cross-warehouse oversight | Configure SLA rules, view regional reports |
| Finance | Billing reconciliation | Configure billing rates, export billing data |
| Sales / Client Servicing | Client relationship | Read-only order and inventory visibility |
| Client User | External client visibility | View own inventory, orders, SLA metrics |
| IT Administrator | System configuration | Full system access |

# 3. Data Models & Database Schema

All tables use **PostgreSQL**.

Key rules:

- UUIDs used for primary keys
- Timestamps stored in UTC
- Soft deletes using `is_active`
- Master data never hard deleted

# 4. API Design

All APIs are **RESTful over HTTPS**.

Response format:

```
{
 success: boolean,
 data: object | array,
 meta: { page, total },
 error: { code, message }
}
```

Versioning:

```
/api/v1/
```

## Order Lifecycle (FSM)

State transitions:

```
new → received → allocated → picked → packed → dispatched → delivered
```

Additional transition:

```
any → cancelled (before dispatch)
```

Partial fulfillment allowed.
